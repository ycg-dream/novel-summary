# 长篇小说结构化摘要工作流

> 把"读 537 万字小说"和"写摘要"两个任务解耦：前者由 Python 脚本做原子化切分 + 子 Agent 独立读全文生成；后者由子 Agent 在充足上下文中输出，主 Agent 只做进度维护与质量校验。

## 适用场景

- **典型输入**：用户给一个中文长篇小说 txt（5 万字以上），要求"全书摘要"/"速看版"/"总结"/"梗概"/"分章节情节"
- **不适用**：用户只要某几章的精简概览（直接 Read 即可）；用户已经提供拆好的 part 文件直接摘要；用户希望保留原句引用而非结构化总结

## 工作流总览

```
[0] **文本诊断**（必做）── diagnose_novel.py 检查编码/章节/广告/屏蔽词/谐音字
        ↓
[1] 智能体读小说头尾判断类型
        ↓
[1.5] **创建独立项目目录**（必做） ── `H:\搜书吧\<书名简化>_摘要\`
        ↓ 所有产出文件（切分/摘要/进度笔记/汇总/脚本）都放此目录
        ↓ 避免污染源小说目录（详见 _learned_patterns.md 主题 27）
        ↓
[2] Python 切分脚本：~20 万字/段，章节边界优先
        ↓
[3] 子 Agent 派发器：每个 part 派一个 general-purpose 子 Agent
        ↓
[4] 子 Agent 独立读 20 万字原文 → 输出结构化摘要（v1）
        ↓
[5] 主 Agent 质量校验 → 通过则进入汇总；不通过则反馈子 Agent 重做
        ↓
[6] 合并所有 part 摘要 → 全书汇总文件
        ↓
[7] **清理 27 个 part 临时文件**（每份 ~600KB，537 万字小说共 ~16MB）  ⚠️ 不要忘记
        ↓
[8] **进度笔记**（强烈推荐）── `<书名>_进度笔记.md` 跟踪跨会话状态
        ↓ 模板见 `templates/progress_template.md`
        ↓
[9] **经验评估 + 沉淀**（本次新增）── 主 Agent 主动归纳"跨书复用"经验
        ├─ 评估清单：`templates/experience_evaluation.md`
        ├─ 累积库：`_learned_patterns.md`
        └─ 单书经验：项目目录 `_经验总结_<书名>.md`
```

## 进度笔记模板

跨会话跟踪摘要进度的标准模板：`templates/progress_template.md`

**必填字段**：
- 第 0 节：基本信息（源文件、字数、卷数、part 数、编码、章节结构）
- 第 1 节：小说类型与风格
- 第 2 节：命名规范（避免跨 part 不一致）
- 第 3 节：R18 内容处理策略（保守/中庸/宽松三选一）
- 第 6 节：进度追踪（已完成/待办/质量校验）
- 第 7 节：已知上下文（子 Agent 提示词复用）
- 第 9 节：下次会话接管指南

**使用方式**：
1. 处理新小说时，复制 `templates/progress_template.md` → `<书名>_进度笔记.md`
2. 填入第 0-3 节（基本信息 + 命名规范 + R18 策略）
3. 每完成一个 part，更新第 6 节"已完成 part" + 质量校验结果
4. 每完成 5-10 个 part，更新一次第 7 节"已知上下文"
5. 新会话接管时，**先读进度笔记 + 已完成 part 摘要**

## R18 内容处理策略（**质量优先 + 升级版优先**）

> ⚠️ **核心原则**：**质量优先**。先派升级版（剧情完整），不行再降级到保守版，**不要默认派保守版"保险"**。

处理 R18 题材时，主 Agent 应按以下顺序选择策略：

| 优先级 | 策略 | 描述 | 触发拦截风险 | 适用场景 |
|--------|------|------|--------------|----------|
| 1️⃣ **默认** | **升级版** | 保留对剧情承启有关键作用的 R18 描写（1 段 50-100 字） | 中-高 | **R18 后宫/黑道/系统流/医生类推荐** |
| 2️⃣ 备选 | 中庸版 | 剧情因果压缩 + 谐音字回避 | 中 | 通用 R18 |
| 3️⃣ 兜底 | 保守版 | 敏感词硬删，可能丢剧情 | 低 | 升级版触发拦截后的单 part 重做 |
| ❌ 不推荐 | 跳过 | API 拦截 4+ 次仍失败 | — | 极少数情况：5 种策略都失败 |

**升级版核心原则**：R18 描写服务于人物弧光与剧情因果。

- ✅ **保留**（用 1 段 50-100 字描述具体动作/结果）：
  - 角色用什么手段制服对方（影响后续反杀/觉醒）
  - 男女主首次结合（情感转折点）
  - 角色"如何被改造"（影响后续人物弧光）
  - 角色"如何对敌人进行情报榨取"（剧情转折）
- ❌ **压缩**（1-2 句概括即可）：
  - 单纯的群交描写、工具/姿势堆砌
  - 与后续剧情无关的 N 人轮奸堆砌（如"接待了 X 名男子"一笔带过）
  - 重复的 SM 套路
  - 与人物塑造无关的纯感官描写

**安全边界（升级版）**：
- 避开高危词：入/操/吊/草/肏/屌
- "轮奸" → 改用 "被多人凌辱" / "遭集体施暴"
- 避开具体器官描写——用"性行为"等概括词
- 避开重复出现的 SM 工具名称堆砌

**"质量优先"的具体执行**：
1. 派子 Agent 时**默认写"升级版"**（不是"保守版"）
2. 子 Agent 触发 API 拦截时：**只重派那个 part 改用保守版**，其他 part 不动
3. 合并时标注"⚠️ 此 part 用保守版生成"
4. **不要因为 1 个 part 触发拦截就全局降级**

## API 拦截兜底流程

派子 Agent 派发某 part 时，子 Agent 返回 `output new_sensitive (1027)` 等 API 拦截错误：

1. **第 1 次**：换更保守的提示词重试（保守版措辞）
2. **第 2 次**：将该 part 拆分为前后两半分别重试
3. **第 3 次**：用"扫读+跳过 R18 段落"策略重试
4. **第 4+ 次**：**接受跳过**——写一个"跳过标记文件"占位：

   ```markdown
   # 【part_NNN_summary】跳过（API 拦截）
   ⚠️ 本 part 因 R18 密度过高触发 MiniMax API 拦截 4 次，无法生成完整摘要
   位置：全书第 N / M 个 part，对应原书第X卷
   跳过原因：尝试 4 种不同策略均失败
   后续方案：人工补做（用户自行阅读后写 1-2 段剧情概述）
   ```

5. **在合并汇总时**：用"⚠️ part N 跳过"标注，并提示用户可人工补做

**注意**：对于 R18 密度特别高的 part，**5 种策略都可能失败**——这是 MiniMax API 的硬性限制，无法绕过。

## 阶段 0：文本诊断（**必做**）

中文网络小说 txt 经常有问题（编码错乱、广告/书友群、屏蔽词、谐音字、章节缺失），**切分前必须先用 diagnose_novel.py 诊断**。

```bash
python ~/.claude/skills/novel-summary/diagnose_novel.py --path /path/to/novel.txt
# 或自动找当前目录的 .txt
python ~/.claude/skills/novel-summary/diagnose_novel.py
```

### 诊断项

| 检查项 | 常见问题 | 修复方式 |
|--------|----------|----------|
| **编码/BOM** | GBK/UTF-8/UTF-16 混杂 | 用 errors='replace' 容错读取 |
| **章节结构** | 缺失/重复/卷号不规范 | 报告缺失章节号，子 Agent 据此调整 |
| **广告/推广** | 关注公众号/书友群/下载 APP | 切分后用 Python 清理 |
| **屏蔽词** | 方块字符/XXX/和谐 | 切分前先识别并标记 |
| **谐音字** | "草"=操、"曰"=日 | **子 Agent 提示词中明确"识别语境"** |
| **错别字** | 的地得混用 | 子 Agent 据此评估 |
| **重复/灌水** | 同一行重复 5+ 次 | 通常是章节标题，**不处理** |

### 诊断报告样例

```
=== 文本质量诊断 ===
文件: novel.txt
  大小: 4,193,275 字节
  编码: gbk
  字符数: 2,127,933
  ✓ 无编码错误

=== 2. 章节结构 ===
  章节数: 764
  卷数: 5
  章节号范围: 1 ~ 297
  ⚠ 缺失章节号 (18 个): [101-109, 201-209]

=== 6. 谐音字/同音字检测 ===
  ⚠ "入" → 疑似"日": 1389 处
  ⚠ "草" → 疑似"操": 238 处
  ⚠ "操" → 疑似"肏": 78 处
  ⚠ "吊" → 疑似"屌": 46 处

=== 10. 推荐切分粒度 ===
  推荐: 20 万字/段（10-50 个 part）
```

### 诊断结果的子 Agent 提示词注入

把诊断结果（**特别是缺失章节 + 谐音字列表**）原样插入子 Agent 提示词的"已知上下文"部分：

```python
# 切分前自动读取 diagnose 报告，注入提示词
diagnose_report = open("diagnose_report.md").read()
prompt += f"""
**小说文件诊断报告**（已运行 diagnose_novel.py）：

{diagnose_report}

请注意：
1. 缺失章节号 [101-109, 201-209] 可能是分卷/外传/附录，请按原文实际顺序处理
2. "入" 在 1389 处出现，但多数是正常词（进入/收入等），仅在"插入/进入（动词）"语境下才是"日"的替代
3. "草" 在 238 处出现，但也是网络常用感叹词，仅在情色/贬低语境下才是"操"的替代
4. 子 Agent 需根据上下文判断真实意图，不要看到"草"就一律解读为"操"
"""
```

## 阶段 1：判断 + 切分

### 1.1 编码识别
```bash
# 检测编码（小说常用 GBK / UTF-16 / UTF-8）
python -c "
import os
path = '<小说路径>'
with open(path, 'rb') as f:
    head = f.read(2)
if head == b'\xff\xfe': enc = 'utf-16-le'
elif head == b'\xfe\xff': enc = 'utf-16-be'
else: enc = 'utf-8'
with open(path, 'r', encoding=enc) as f:
    print('字符数:', len(f.read()))
"
```

### 1.2 切分粒度决策
| 总字数 | 切分粒度 | part 数 | 适用 |
|--------|----------|---------|------|
| < 30 万字 | 不切分 | 1 个 part | 主 Agent 直接读 |
| 30-200 万字 | 10-20 万字/段 | 2-20 | 主 Agent 可监督 |
| **200-1000 万字** | **20 万字/段** | **10-50** | **推荐用子 Agent** |
| > 1000 万字 | 30 万字/段 | 30+ | 子 Agent 仍可行，但每 part 需更长 |

**关键经验**：
- 切分粒度**不要太小**（如 3500-5000 字/段）——会导致摘要碎片化丢失剧情弧
- 切分粒度**不要太大**（> 30 万字/段）——子 Agent 容易超时
- **20 万字 ≈ 50-65 章** 是甜点：能看到完整故事弧 + 单 Agent 能在 5 分钟内完成

### 1.3 切分脚本（参考 split_novel2.py）
```python
# 关键参数
TARGET_PART_SIZE = 200_000  # 20 万字
# 切分时优先在章节标记处切（中文网络小说常用"第NNNN章"或"第N卷"）
# 若章节不规整，按自然段落（空行）+ 字符数就近切
# 不要在句子中间硬切
```

## 阶段 2：子 Agent 派发（核心创新）

### 2.1 为什么用子 Agent

**主 Agent 串行读的致命问题**：
- 主 Agent 每读一个 part 就把内容塞进自己的上下文
- 几个 part 后**主 Agent 上下文爆掉**（537 万字任务实测主 Agent 撑不住 5 个 part）
- 主 Agent 失去耐心，摘要质量**逐 part 下降**

**子 Agent 方案**：
- 每个子 Agent **独立读 1 个 part 的 20 万字原文** + **独立输出摘要**
- 子 Agent 输出**返回主 Agent 后可丢弃**（不占主 Agent 上下文）
- 主 Agent 只需维护一份**进度笔记**（共享已知上下文）

### 2.2 子 Agent 提示词模板

```python
prompt = f"""
你的任务：为《{book_name}》这本长篇小说的**第 {{n}} 个片段**生成一份高质量中文摘要。

**输入文件**：`{{input_path}}`（约 20 万字，UTF-8，覆盖第 {{chap_start}}-{{chap_end}} 章）

⚠️ **注意**：本任务**不是**接续前面的摘要，而是**首次独立生成**第 {{n}} 个 part 的摘要。
你**不需要**依赖前文——直接读 {{input_path}} 原文，根据你读到的内容写摘要。

**摘要要求**（6 大模块）：
1. **核心主线**（200-300 字）：用一段话交代本片段的主要剧情走向
2. **关键剧情节点**（按出现顺序 8-15 个，每条 80-150 字，必须有具体人物名/对话/决策/转折）
3. **新增人物**（首次出现或地位提升，注明身份/能力/与主角关系）
4. **世界观/设定扩充**（新规则/新势力/新科技/新概念）
5. **政治格局变化**
6. **伏笔/未解线索**

**风格要求**：
- 第三人称客观叙述
- 直接、密度高、避免空话
- 关键对话可**直接引用原文短句**（不超过 20 字）
- **严禁编造**——如果某段文本你没读懂，宁可省略也不要瞎猜
- **加料章（标题以 ❤ 开头）**也要标注，但**不写加料内容**
- 字符控制在 **2000-3000 中文字**

**输出格式**：
- 写入文件 `{{output_path}}`
- 顶部写"【part_{{:03d}}_summary】...
- 用 `## 一、` `## 二、` 等数字编号

**不要做的事**：
- 不要写"前情回顾"
- 不要使用情绪化语言
- 不要超过 3500 字
"""
```

### 2.3 已知上下文怎么给子 Agent

**错误做法**：让主 Agent 把前 20 个 part 的摘要复制到每个子 Agent 提示词里——很快提示词会超过 100K token。

**正确做法**：
1. **主 Agent 维护一份 `<book_name>_progress.md`**：精炼的"已知上下文"（< 5K token）
2. 每个子 Agent 提示词**前 5-10 个 part 时** 附"已知上下文"段落
3. 之后每 5-10 个 part 增量更新一次

### 2.4 ⚠️ 子 Agent 禁止写"承接上文"内容（2026-06-10 用户反馈）

子 Agent **看不到前 part 的内容**——它写"承接上文 part X"等于凭空编造。

**正确分工**：
- **子 Agent**：只摘要本 part 内部的剧情（节点、人物、设定）
- **主 Agent**：合并时**自己写过渡句**（基于已读的所有 part 摘要）

**子 Agent 提示词应禁止**：
- ❌ "承接上文 part X"
- ❌ "延续前 part"
- ❌ "如前文所述"
- ❌ 任何引用前 part 内容的话

**子 Agent 提示词应明确写**：

```
⚠️ 你**只能看到本 part 的内容**——不要写任何引用前 part 内容的话
（如"承接上文""如前文所述""延续前 part"等）。
你只摘要本 part 内部的剧情，跨 part 衔接由主 Agent 负责。
```


## 阶段 3：质量校验（主 Agent 责任）

### 100 分制评分体系（2026-06-09 更新）

**核心原则**：**内容质量权重最高（60%），字数/格式权重最低（10%）**。

主 Agent 必须**亲自 Read** 子 Agent 输出的 summary 文件，按 100 分制评分。

| 维度 | 权重 | 评分要点 |
|------|------|----------|
| **内容质量** | **60 分** | 节点是否具体（人物名+对话+决策+转折）？是否覆盖核心剧情？是否避开瞎猜/编造？ |
| **逻辑清晰度** | **20 分** | 节点间时序/因果关系？人物关系是否清楚？ |
| **还原性** | **10 分** | 人物名/设定/地点是否与原文 100% 一致？ |
| **字数/格式** | **10 分** | 字数在目标范围（2000-3500）？6/7 模块齐全？Markdown 规范？ |

**通过标准**：≥ 80 分通过 / < 80 分反馈重做（最多 1 次重试）。

详细评分标准见 `_learned_patterns.md` 主题 10。

### 主 Agent 审核输出格式

```
part_NNN 审核完成
- 内容质量: XX/60
- 逻辑清晰度: XX/20
- 还原性: XX/10
- 字数/格式: XX/10
- **总分: XX/100**
- 通过/不通过
- 备注：（如有质量问题）
```

### 不通过怎么办

```python
# 反馈子 Agent 重做
Agent(prompt=f"""
你之前的 part {n} 摘要评分 X/100，未通过。问题：
1. （内容质量问题：节点不够具体/有编造嫌疑）
2. （逻辑/还原性问题）

请重读 {{input_path}} 原文，重新撰写 2000-3500 字摘要，严格遵守 6 大模块。
特别注意：
- 节点必须具体（人物名+对话+决策+转折）
- 避免空话/编造
- 人物名/设定忠实于原文
输出到 {{output_path}}。
""")
```

## 阶段 4：合并

### 4.1 合并脚本
```python
# merge_summaries.py 模板
for n in range(1, N+1):
    if os.path.exists(f'part_{n:03d}_summary.md'):
        with open(f'part_{n:03d}_summary.md', 'r', encoding='utf-8') as f:
            out.write(f.read().strip() + '\n\n---\n\n')
```

### 4.2 输出文件命名
- 单 part 摘要：`part_NNN_summary.md`
- 全书汇总：`<book_name>_全书摘要汇总.md`

## 阶段 6：经验评估 + 沉淀（本次新增）

完成全书摘要 + 合并 + 清理 + 验证后，主 Agent **主动**做一次经验评估：

**评估工具**：`templates/experience_evaluation.md`（含 9 项必查清单）

**评估标准**：
- **A 类（跨书复用）**：不依赖具体书名/人物/事件，下次处理任何小说都能用
- **B 类（单书特定）**：依赖具体书名/人物/事件，仅本书适用

**沉淀方式**：
- A 类 → 自动追加到 `_learned_patterns.md`（按主题组织，避免冗余）
- B 类 → 留在项目目录的 `_经验总结_<书名>.md`

- 主题 7：进度笔记跨会话接续

- 主题 9：开放式结局处理

**下次新会话时**：主 Agent 先读 `_learned_patterns.md`，把适用经验注入子 Agent 提示词——避免重复踩坑。

## 关键经验教训

### ✅ 必须做
1. **20 万字/段**——不要切太细也不要切太粗
2. **子 Agent 独立读全文**——主 Agent 不读原文
3. **6 大模块结构化**——保证风格一致
4. **进度笔记共享上下文**——避免子 Agent 不知道前置设定
5. **严禁编造 + 加料章标注**——质量底线
6. **质量校验**——主 Agent 抽样验证子 Agent 输出

### ❌ 不要做
1. ❌ 主 Agent 串行读 20 个 part——上下文会爆
2. ❌ 切 3500-5000 字/段——摘要碎片化
3. ❌ 给子 Agent 复制所有前文摘要——提示词爆炸
4. ❌ 跳过质量校验直接汇总——错误会传播
5. ❌ 期望子 Agent 一致字数——它们容易超 1.5-2 倍（**接受超长，质量更优**）
6. ❌ 试图一次性跑完——主 Agent 会爆，按 4-8 个 part/会话 接力

### 性能数据（实测，537 万字小说）
- 主 Agent 串行 5 个 part → 上下文爆炸
- **子 Agent 串行 27 个 part → 0 上下文压力，6 个会话接力**
- 单 part 处理：3-7 分钟
- 质量提升：v1 (主 Agent) vs v2 (子 Agent) → 中文字数 ×3，关键事件 ×2-3

## 排错指南

| 症状 | 原因 | 修复 |
|------|------|------|
| 编码检测说"UTF-8"但实际是 GBK | 头部 4KB 是 ASCII 标题 | 用 multi-offset 检测（已内置） |
| **子 Agent 把"草"=操、"曰"=日 一律解读** | 没看到诊断报告 | **跑 diagnose_novel.py + 把谐音字列表注入提示词** |
| 子 Agent 输出 `output new_sensitive` 错误 | part 含敏感内容 | 重写提示词，强调"只写剧情，不写加料内容" |
| 子 Agent 写"前情回顾" | 没看到"不要写前情" | 在提示词首段加 ⚠️ 警告 |
| 子 Agent 摘要太短（< 1500 字） | 读得不够 | 提示词加"必须 ≥ 10 个关键事件，每条 ≥ 80 字" |
| 主 Agent 上下文爆 | 累计太多 part | 立即换新会话，从中断 part 继续 |
| 跨 part 人物名不一致 | 子 Agent 不知道前置 | progress.md 同步更新人物名表 |
| 加料章被忽略 | 子 Agent 主动跳过 | 提示词强调"加料章用 ❤ 标记即可，不写内容" |
| 章节摘要把广告/书友群也写进来 | 切分前未清理 | 跑 diagnose 识别，再 Python 切掉 |
| **磁盘占用大（十几 GB）** | **合并后忘了清理 part 临时文件** | **运行 `python cleanup_parts.py` 删除** |
