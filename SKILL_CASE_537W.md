## 摘要生成方案

### 第一阶段：主 Agent 手做（part 1-5）

**问题**：
- 主 Agent 直接 Read 原文 → 上下文累积爆炸
- 5 个 part 后主 Agent 已显疲态，part 5 质量明显下降
- 单 part 仅写 1000-1200 字（信息密度低）

**已删除**（但 work 仍保留在 v1 文件里）。

### 第二阶段：尝试 6 大模块结构化（v1 重做）

意识到质量差距后，**用子 Agent 串行处理 part 6-27**：
- 派 `Agent` 工具调用 `general-purpose` 子 Agent
- 每个子 Agent 独立读 20 万字原文
- 子 Agent 输出 6 大模块结构化摘要
- 主 Agent 维护 `witch_progress.md` 进度笔记
- 主 Agent 提示词含"严禁编造 + 加料章标注 + 关键对话引用"

**结果**：
- v1 摘要（part 6-27）：中文字数 2000-5500，**质量明显优于主 Agent 手做**
- 关键事件数 ×2-3
- 伏笔记录 ×3
- 加料章标注齐全

### 第三阶段：质量升级（v2 重做）

意识到 v1（主 Agent）和 v2（子 Agent）质量差距后，**重做 part 1-5**：
- 用 v1 提示词（同 v2 模板）派子 Agent 重做
- part 4 第一次失败（API 错误 `output new_sensitive`）——**加料章内容触发敏感词**
- part 4 第二次：改提示词强调"只写剧情，不复述加料内容"
- 5 份 v2 摘要全部完成

## 文本质量诊断（实战新增）

`diagnose_novel.py` 是基于 213 万字《这个刺客有毛病》实战经验新增的诊断工具。

**典型 txt 常见问题**（**这个刺客有毛病**诊断报告）：

```
=== 2. 章节结构 ===
  章节数: 764
  卷数: 5
  章节号范围: 1 ~ 297
  ⚠ 缺失章节号 (18 个): [101-109, 201-209]

=== 6. 谐音字/同音字检测 ===
  ⚠ "入" → 疑似"日": 1389 处
  ⚠ "草" → 疑似"操": 238 处
  ⚠ "操" → 疑似"肏": 78 处
  ⚠ "吊" → 疑似"屌": 46 处

=== 10. 推荐切分粒度 ===
  推荐: 20 万字/段（10-50 个 part）
```

**关键发现**：
- 缺失章节号（101-109、201-209）——这两组章节号是分卷/外传/附录，要按原文如实记录
- 谐音字 1300+ 处（"入"、"草"）——子 Agent 需结合语境判断真实意图
- 误报控制：避免"加神秘与微"被误判为"联系方式"（含"加"和"微"）

**修复策略**：
- 诊断结果注入子 Agent 提示词的"已知上下文"部分
- 让子 Agent 明确："入"多数情况是正常词（进入/收入），仅在"插入/进入（动词）"语境下才是"日"的替代
- 子 Agent 提示词必须包含完整的诊断报告（**不能只说"注意谐音字"——必须列具体词和出现次数**）

## 对比：与 DeepSeek 建议的方案

| 维度 | DeepSeek 原方案 | 实际执行 |
|------|----------------|----------|
| 切分粒度 | 3500-5000 字/段 | **20 万字/段**（更优） |
| 模型 | MiniMax abab6.5s-chat API | Claude Code 子 Agent（零成本） |
| 调用方式 | API 批处理 | Agent 工具派发 |
| 上下文压力 | API 独立 | 主 Agent 不读原文 |
| 成本 | ¥5-15 API | **0 元** |
| 实际耗时 | 估计 10-20 分钟 | 实际 6-8 小时（5-6 会话） |

**关键改进**：
1. 切分粒度从 3500 字提升到 20 万字——避免摘要碎片化
2. 用 Claude 子 Agent 替代 API——零成本+免配置
3. 进度笔记（progress.md）共享上下文——避免提示词爆炸
4. v2 升级机制——发现主 Agent 质量差时主动用子 Agent 重做

## 经验教训（写进 SKILL.md）

### ✅ 必须做
1. 20 万字/段（不要更小也不要更大）
2. 子 Agent 独立读全文（主 Agent 不读）
3. 6 大模块结构化提示词
4. 进度笔记共享上下文
5. 严禁编造 + 加料章标注
6. 质量校验（v1→v2 升级机制）

### ❌ 不要做
1. 主 Agent 串行读 5+ 个 part（上下文爆）
2. 切 3500-5000 字/段（摘要碎片化）
3. 复制前文摘要到子 Agent 提示词（提示词爆炸）
4. 跳过质量校验（错误传播）
5. 期望子 Agent 字数一致（接受超长）
6. 一次性跑完全部（按 4-8 个 part/会话 接力）
