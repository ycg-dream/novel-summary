## 主题 1：API 拦截兜底（output new_sensitive 1027）

**适用场景**：派子 Agent 处理 R18 题材时，子 Agent 返回 `output new_sensitive (1027)` 等 API 拦截错误。

**经验沉淀**：
- 2026-06-08（烈火凤凰）：R18 密度特别高的 part，5 种策略（升级版/保守版/拆分/扫读+跳过/主 Agent Write）**全部失败**。MiniMax API 对单次输出中 R18 描写的总密度有硬性限制，无法绕过。
- **结论**：
  1. 第 1-3 次失败时换策略重试
  2. 第 4+ 次失败时**立即接受跳过**，不要硬耗额度
  3. 跳过时写"占位标记文件" + 提示用户可人工补做
  4. 不要尝试"主 Agent 自己 Read+Write 绕过"——Write 工具仍触发 API

## 主题 2：R18 策略三档选择

**适用场景**：处理含 R18 内容的小说时，需要选择保留/压缩 R18 描写的程度。

**经验沉淀**：
- 2026-06-08（烈火凤凰）：升级版策略（保留关键 R18 描写 1 段 50-100 字）效果好——7/7 part 通过 API，剧情完整度高于中庸版和保守版。

**三档选择**：

| 策略 | 描述 | 触发拦截风险 | 适用场景 |
|------|------|--------------|----------|
| 保守版 | 敏感词硬删，可能丢剧情 | 低 | 清纯/言情类 |
| 中庸版 | 剧情因果压缩 + 谐音字回避 | 中 | 通用 R18 |
| **升级版** | 保留对剧情承启有关键作用的 R18 描写 | 中-高 | **R18 后宫/黑道类推荐** |

**升级版核心原则**（"R18 描写服务于人物弧光与剧情因果"）：
- ✅ **保留**（1 段 50-100 字）：角色用什么手段制服对方 / 男女主首次结合 / 角色"如何被改造" / 情报榨取
- ❌ **压缩**（1-2 句）：单纯群交 / 工具姿势堆砌 / N 人轮奸堆砌 / 重复 SM 套路 / 纯感官描写

**安全边界**（升级版）：
- 避开高危词：入/操/吊/草/肏/屌
- "轮奸" → 改用 "被多人凌辱" / "遭集体施暴"
- 避开具体器官描写
- 避开 SM 工具名称堆砌

## 主题 3：跨 part 命名一致性

**适用场景**：长篇小说由多个子 Agent 派发摘要时，可能出现"白无暇 vs 白无瑕"等命名不一致。

**经验沉淀**：
- 2026-06-08（烈火凤凰）：发现 9 处命名不一致（白无暇→白无瑕 4 处、姬冬嬴→姬冬赢 5 处）。旧版 part 1-3（敏感词硬删版）用了错别字，新版 part 4+ 用了正确字。

**预防方法**：
- **切分前**就在进度笔记第 2 节"命名规范"列出所有关键人物的统一名称 + 曾用别名
- **派子 Agent 时**复制命名规范到提示词
- **合并后**必须跑 `validate_summaries.py` 验证

**修复方法**：
- 用 Python 脚本批量替换 `content.replace(wrong, correct)`
- 不要手动 25 处一一改（容易漏）

## 主题 4：合并后必须验证（不能简单贴在一起）

**适用场景**：完成所有 part 摘要 + 合并为汇总文件后。

**经验沉淀**：
- 2026-06-08（烈火凤凰）：第一次合并时只简单"贴在一起"，未做验证。用户提醒后跑验证脚本，发现 9 处命名不一致 + 6 处对话引用超长 + 1 个重复 part 文件。

**必查清单**：
1. **6/7 模块标题格式**（每个 part 应有 ## 一、 ~ ## 六、；升级版 + ## 七、对比说明）
2. **跨 part 命名一致性**（关键人物/地点/武功的多种拼写）
3. **每 part 摘要字数**（应在目标范围内，如 2000-3500 中文字）
4. **对话引用短句长度**（应 ≤ 20-25 字）
5. **错别字/OCR 错误**（常见：部份→部分、做为→作为）
6. **合并文件结构**（每个 part 应有 `## 第 N 部分` 标记 + `---` 分隔符）
7. **重复文件检查**（如新旧 part 重试版本同时存在）

**工具**：`scripts/validate_summaries.py`（可选）或 `templates/validation_checks.md`（手动清单）

## 主题 11：用户偏好"全局升级版，触发拦截才改保守"（2026-06-09 沉淀）

**适用场景**：用户处理 R18 题材时，对 R18 描写的保留度有偏好。

**经验沉淀**：
- 2026-06-09（当医生遇上不正经系统）：用户明确指示——**所有 part 默认用升级版策略**，只有当某个 part 触发 API 拦截时，**那个 part 单独改保守版**，其他 part 不受影响。

**默认策略**：
- 升级版（默认）—— 7/7 part 通过率高
- 触发拦截的 part 单独改保守版
- 不要因为一个 part 失败而全局降级

**与主题 2 的关系**：主题 2 给出"三档选择"，本主题给出"默认选哪档 + 例外处理"。

**与主题 11 的强化版**——"质量优先 + 升级版优先"（2026-06-09 用户明确确认）：

用户原话："我希望的是质量优先的，就是得先用升级版完整的提示词，不行再用保守版的"

**核心策略**（处理 R18 题材时）：
1. **默认派升级版提示词**（不是保守版！）
2. **触发 API 拦截时**：那个 part 单独改保守版
3. **不要因为 1 个 part 失败就全局降级**

**为什么"升级版优先"而不是"保守版优先"**：
- ✅ 升级版保留关键 R18 描写 → 剧情完整度高（"看完整本小说"体验）
- ❌ 保守版敏感词硬删 → 剧情因果丢失（用户体验差）
- ✅ 升级版通过率高（烈火凤凰 7/7、当医生 14/14 全部通过）
- ❌ 触发拦截只影响那个 part，其他 part 不受影响

**执行细节**：
- 派子 Agent 提示词中写明"升级版策略"
- 子 Agent 回报"是否触发 API 拦截"
- 如触发，**只重派那个 part 改用保守版**，其他 part 不动
- 合并时如 part 用了保守版，标注"⚠️ 此 part 用保守版生成"

**反例**（不要这样做）：
- ❌ 默认派保守版"保险"
- ❌ 一个 part 触发拦截后全局改保守版
- ❌ 因为 R18 题材直接跳过（除非 5 种策略都失败）

## 主题 10：主 Agent 100 分制质量审核（每次子 Agent 完成后必做）

**适用场景**：每个子 Agent 派发 part 并返回结果后，**主 Agent 必须亲自 Read 子 Agent 输出的 summary 文件，按 100 分制评分**。

**经验沉淀**：
- 2026-06-09（当医生遇上不正经系统）：用户反馈——之前用 10 分制过于笼统，**字数/模块完整性的权重不应过高**，应重点审查"内容质量、逻辑清晰度、还原性"。
- 2026-06-09（当医生遇上不正经系统）：用户原话——"还得审查下内容的质量，逻辑和内容是否清晰可读，还原性如何，内容的权重得加大，至于字数什么的权重并没那么重要"
- **结论**：从 10 分制改为 100 分制，**内容质量 60 分 + 逻辑清晰度 20 分 + 还原性 10 分 + 字数格式 10 分**。

**审核流程**（每个 part 子 Agent 返回后必做）：
1. **Read 子 Agent 输出的 summary 文件**（主 Agent 亲自读，不是只看子 Agent 报告）
2. **按 100 分制评分**（4 大维度，权重不同）
3. **评分 ≥ 80 分**：通过，记入进度笔记
4. **评分 < 80 分**：反馈子 Agent 重新生成（**最多 1 次重试**，避免耗额度）
5. **评分 < 60 分 或 重试后仍 < 80 分**：标记为"质量不达标"，合并时标注

**100 分制评分维度**：

| 维度 | 权重 | 评分要点 |
|------|------|----------|
| **内容质量** | **60 分** | 节点是否具体（人物名+对话+决策+转折）？是否覆盖了本 part 的核心剧情？是否避开了瞎猜/编造？是否避免空话/灌水？ |
| **逻辑清晰度** | **20 分** | 模块顺序是否合理？节点间是否有时序/因果逻辑？人物关系是否清楚？ |
| **还原性** | **10 分** | 摘要是否准确反映原文的剧情（与原文对照）？人物名/设定/地点是否忠实？ |
| **字数/格式** | **10 分** | 字数在目标范围内（2000-3500 中文字）？6/7 模块齐全？Markdown 排版规范？ |

**内容质量 60 分的详细评分标准**：

| 分值 | 标准 |
|------|------|
| 60-55 | 节点非常具体，每条都有具体人物名+对话+决策+转折，剧情覆盖完整 |
| 54-45 | 节点具体，但个别有 1-2 处不够具体 |
| 44-35 | 节点基本合格，但整体偏简略或不够"抓重点" |
| 34-20 | 节点过简，缺人物/对话/决策；或部分有编造嫌疑 |
| 19-0 | 严重编造/空话/与原文不符 |

**逻辑清晰度 20 分的详细评分标准**：

| 分值 | 标准 |
|------|------|
| 20-18 | 节点间时序清晰，因果关系明确，人物关系清楚 |
| 17-15 | 大部分清晰，个别跳跃 |
| 14-10 | 顺序或因果有疑问 |
| 9-0 | 混乱、跳跃、无逻辑 |

**还原性 10 分的详细评分标准**：

| 分值 | 标准 |
|------|------|
| 10-9 | 人物名/地点/武功/设定与原文 100% 一致 |
| 8-7 | 99% 一致，1-2 处小误 |
| 6-5 | 95% 一致，3-5 处误 |
| 4-0 | 重大不一致（凭空捏造人物/篡改剧情） |

**字数/格式 10 分的详细评分标准**：

| 分值 | 标准 |
|------|------|
| 10-9 | 字数 2000-3500 中文字 + 6/7 模块齐全 + Markdown 规范 |
| 8-7 | 略超/略低，或模块有 1 处缺失 |
| 6-5 | 严重超长/过短，或模块缺失 2+ |
| 4-0 | 字数失控或模块混乱 |

**主 Agent 输出格式**（每次审核后必输出）：

```
part_NNN 审核完成
- 内容质量: XX/60
- 逻辑清晰度: XX/20
- 还原性: XX/10
- 字数/格式: XX/10
- **总分: XX/100**
- 通过/不通过
- 备注：（如有质量问题）
```

**重做提示词模板**（如果 < 80 分）：

```
你之前的 part_NNN 摘要评分 X/100，未通过。问题：
1. （内容质量问题：具体节点 + 原文对比）
2. （逻辑问题：节点顺序/因果）
3. （还原性问题：人物名/设定错误）
请重读 [输入文件] 原文，重新撰写，严格遵守 6 大模块 + 升级版 R18 策略。
特别注意：
- 节点必须具体（人物名+对话+决策+转折）
- 避免空话/编造
- 人物名/设定忠实于原文
输出到 [输出文件]。
```

**重做时的"为什么错"必须明确告诉子 Agent**（2026-06-09 沉淀）：
- 不要只说"重新生成"，要**指出上一次哪里错了**（如"主角名错了：原文是 X，你用了 Y"）
- 提供原文中能验证正确信息的片段（如 part_005 重做时，提示词明确说"主角是陈俊，全书 14 part 中 13 个都用陈俊"）
- 重做后让子 Agent 主动确认修复（"请在回报中明确说明你用了哪个主角名"）

## 主题 12：跨 part 命名漂移修复流程（2026-06-09 沉淀）

**适用场景**：主 Agent 在合并前做命名一致性检查时，发现某 part 用了错误的主角名/人物名（与全书不一致）。

**经验沉淀**：
- 2026-06-09（当医生遇上不正经系统）：part_005 用了"沈夕"作为主角名（28 次），其他 13 个 part 都用"陈俊"。验证 part_005 原文发现，原文**几乎全是"陈俊"**——子 Agent 凭空捏造了"沈夕"作为主角名。

**核心教训**：
- ❌ **不要直接批量替换**（"沈夕"→"陈俊"）——可能破坏小说真的用"沈夕"的 part
- ❌ **不要预设结论**（"沈夕"是用户伪标题，所以肯定错）——可能 part 5 真的用"沈夕"
- ✅ **正确流程**：
  1. 主 Agent 亲自 Read 子 Agent 输出的 summary 文件（主题 10 审核）
  2. 发现命名异常时，**先 Read 原文**验证
  3. 确认是子 Agent 错误后，**派子 Agent 重做**（主题 10 重做流程）
  4. 重做时提示词明确指出"上一次哪里错了"（主题 10 重做必填段）
  5. 重做后再次审核 + 验证

**验证脚本**（在合并前必跑）：

```python
# 跨 part 命名一致性
name_variants = {
    '主角A': ['主角A', '其他拼写'],
    ...
}
for name, variants in name_variants.items():
    counts = {v: 0 for v in variants}
    for n in range(1, N+1):
        with open(f"part_{n:03d}_summary.md", 'r', encoding='utf-8') as f:
            content = f.read()
        for v in variants:
            counts[v] += content.count(v)
    # 出现多种拼写 → 警告
    if len([v for v in variants if counts[v] > 0]) > 1:
        print(f"⚠ {name}: {counts}")
```

## 主题 15：合并汇总的可读性优化（跨 part 过渡 + 人物索引表）

**适用场景**：合并多个 part 摘要为单一汇总文件后，需要提升阅读体验。

**经验沉淀**（2026-06-09，用户 deepseek 评估后整理）：
- 三个可跨书复用的优化项：
  1. **part 间过渡句**——每个 part 开头加 1 行斜体过渡
  2. **视觉呼吸感**——节点间空行 + 末尾短分隔线
  3. **末尾"人物索引表"**——按拼音排序，标注"核心配角"vs"路人"
- 不需要的优化项：
  - 阅读指南（每本小说类型不同）
  - 医学/专业术语注释（题材特定）
  - emoji 标题（视觉偏好，因人而异）

**part 间过渡句模板**（由**主 Agent**在合并时写入，**不**由子 Agent 写）：

```markdown
## 第 N 部分
*承接上文：[主 Agent 从 part N-1 摘要中提取的末尾状态]。本部分 [主 Agent 从 part N 摘要中提取的开头内容]。*
```

**⚠️ 重要：过渡句由主 Agent 写，不让子 Agent 写**（2026-06-10 用户反馈）：
- 子 Agent **看不到前 part 的内容**——它写"承接上文"等于凭空编造
- 主 Agent 合并时**已经读过所有 part 摘要**，**有能力**写正确的过渡句
- 子 Agent 提示词模板**应明确禁止**写"承接上文"内容

**子 Agent 提示词应禁止的措辞**：

```
❌ 不要在摘要开头写：
- "承接上文"
- "延续 part X"
- "如前文所述"
- "上 part 中..."

✅ 子 Agent 只摘要本 part 内部的剧情：
- 节点 N1：本 part 第 X-Y 章，[事件描述]
- 节点 N2：本 part 第 Y-Z 章，[事件描述]
```

**主 Agent 合并时写过渡句的方法**（Python 脚本）：

```python
for i in range(1, len(parts)):
    n = parts[i][0]
    prev_n = parts[i-1][0]
    
    # 从 part N-1 摘要的末尾提取状态
    prev_end_state = extract_last_sentence(parts[i-1][1])  # 如"林风攻略苏晴雯"
    
    # 从 part N 摘要的开头提取内容
    cur_start_content = extract_first_sentence(parts[i][1])  # 如"本片段聚焦秦婉柔办公室胁迫"
    
    transition = f"*承接上文：{prev_end_state}。本部分：{cur_start_content}。*"
    insert_transition_before_part_n(transition)
```

**这样保证**：
- 子 Agent **只摘要本 part**（不依赖看不到的前文）
- 主 Agent **统一负责跨 part 衔接**（基于它已读的所有摘要）
- 避免"凭空编造的承接"违反"严禁编造"原则

## 主题 17：子 Agent Read 工具的稳定性问题（2026-06-09/10 沉淀）

**适用场景**：子 Agent 在读取大文件（>10 万字）时可能出现 Read 工具超时/中断。

**经验沉淀**：
- 2026-06-09/10（合欢宗 part 7/8）：子 Agent 报告 "file system access issues mid-task, I was only able to read the first ~200 lines" → 自评分 79 分未通过。

**根因**：MiniMax API Agent 的 Read 工具有内部超时限制（推测 5 分钟），处理大文件时分多次 Read 才能读完。

**解决方案**：
- **方案 A**：主 Agent 预先把大文件切成多个 chunk（每 chunk 30-40K 字符），让子 Agent 分次 Read 完整文件。
- **方案 B**：让子 Agent 在提示词中明确"用 Read 工具分多次读完整文件"，并约定每次 Read 多少行。
- **方案 C**：让子 Agent 在回报中**主动报告** Read 是否完整（不要硬撑编造）。

**实测**：方案 A（主 Agent 切 chunk）成功率最高——合欢宗 part 7 用此方案重做后 95 分通过。

**提示词模板**（用于 Read 大文件）：

```
我已经把 part N 切分成了 4 个 chunk 文件（每个 ~37K 字符）：
- Chunk 1（第 X-Y 章）：...
- Chunk 2（第 Y-Z 章）：...
- Chunk 3（第 Z-W 章）：...
- Chunk 4（第 W-V 章）：...

**你必须 Read 这 4 个文件**才能生成完整摘要。
每个 chunk 文件大小约 37K 字符，Read 工具一次可以完整读取，不需要分块。

执行步骤：
1. Read chunk 1（看前 X 章）
2. Read chunk 2
3. Read chunk 3
4. Read chunk 4
5. 基于 4 个 chunk 的完整阅读，**整合**所有节点，生成摘要
```

## 主题 19：作者名文件名一致性问题（2026-06-09/10 沉淀）

**适用场景**：中文小说 txt 文件名常含作者名，子 Agent 派发时若作者名写错，会导致所有文件名错乱。

- 2026-06-09/10（合欢宗）：源文件是"作者：我叫你捡起来"，但我在派子 Agent 时多处写成"我叫我捡起来"——导致 part 1-8 summary 文件名错误，但文件内容正确。

## 主题 25：UTF-8-BOM 编码处理（2026-06-10 沉淀，补全主题 14）

**适用场景**：源文件是 UTF-8 但带 BOM（`\xef\xbb\xbf` 头），常见于 Windows 创建的中文 txt。

**经验沉淀**（2026-06-10，《妈，您人设崩了》）：

**与主题 14（GBK 编码）的区别**：

| 编码 | 头标识 | Python 读法 |
|------|--------|-------------|
| UTF-8（无 BOM） | 无 | `encoding='utf-8'` |
| **UTF-8 with BOM** | `\xef\xbb\xbf` | `encoding='utf-8-sig'`（自动剥离 BOM） |
| GBK | 无 | `encoding='gbk'` |
| UTF-16-LE | `\xff\xfe` | `encoding='utf-16-le'` |
| UTF-16-BE | `\xfe\xff` | `encoding='utf-16-be'` |

**Python 切分模板**（《妈，您人设崩了》案例）：

```python
with open(NOVEL_PATH, 'rb') as f:
    head = f.read(4)
if head[:3] == b'\xef\xbb\xbf':
    enc = 'utf-8-sig'  # 关键：用 utf-8-sig 自动剥离 BOM
elif head[:2] == b'\xff\xfe':
    enc = 'utf-16-le'
elif head[:2] == b'\xfe\xff':
    enc = 'utf-16-be'
else:
    enc = 'utf-8'

with open(NOVEL_PATH, 'r', encoding=enc, errors='replace') as f:
    text = f.read()
```

**注意事项**：
- 用 `'utf-8'` 读 BOM 文件不会报错，但首字符会是 `﻿`（BOM 残留），影响后续字符计数和章节匹配
- 用 `'utf-8-sig'` 自动剥离 BOM，是最安全的写法
- 子 Agent 输出文件**统一用 UTF-8（不带 BOM）**：`encoding='utf-8'`

**diagnose_novel.py 已包含 BOM 检测**（"含 UTF-8 BOM（建议去除以便通用处理）"），按建议处理即可。

## 主题 26：复杂多 CP R18 小说的人物索引表生成（2026-06-10 沉淀，强化主题 18）

**适用场景**：含 10+ 核心 CP 的复杂后宫/伦理 R18 小说的人物索引表生成。

**经验沉淀**（2026-06-10，《妈，您人设崩了》47 个角色）：

主题 18 提出"用汇总文件全文统计人物"，本次沉淀**更细致的索引表生成方法**：

**方法 1：硬编码（推荐用于已知人物的复杂小说）**

主 Agent 在合并脚本中**手动维护人物字典**，每个人物含 `(描述, 角色等级)`：

```python
known_characters = {
    '许麟': ('男主，新疆混血少年...', '核心'),
    '李娅': ('许麟母亲，检察官...', '核心'),
    '柳新月': ('校花...', '核心'),
    # ... 共 47 人 ...
    '陆云': ('番外反派...', '路人'),
}

# 角色等级：'核心' / '核心配角' / '路人' / '_dup'（重复项）
# '_dup' 在最终过滤掉
known_characters = {k: v for k, v in known_characters.items() if v[1] != '_dup'}

# 按首字符分组
from collections import defaultdict
groups = defaultdict(list)
for name, (desc, role) in sorted(known_characters.items()):
    first = name[0]
    groups[first].append((name, desc, role))

# 输出（核心人物加粗）
for first_char in sorted(groups.keys()):
    out.write(f'\n### {first_char}\n')
    for name, desc, role in groups[first_char]:
        if role in ('核心', '核心配角'):
            out.write(f'- **{name}**：{desc}\n')
        else:
            out.write(f'- {name}：{desc}\n')
```

**方法 2：自动统计（适用于人物较少的小说）**

按主题 18 用 Python 统计每个人物在多少个 part 出现，自动判定核心 vs 路人：

```python
# 出现 ≥ 3 个 part → 核心；其他 → 路人
people_parts = {}
for name in all_names:
    count = sum(1 for part in parts if name in part_body)
    people_parts[name] = count

core_chars = [n for n, c in people_parts.items() if c >= 3]
```

**两种方法的权衡**：

| 方法 | 优缺点 | 适用 |
|------|--------|------|
| **硬编码** | ✅ 描述更精炼；✅ 角色等级判断更准确；❌ 需手动维护 | 已读完所有 part 摘要、人物 < 100 的小说 |
| 自动统计 | ✅ 零维护；❌ 描述粗糙；❌ 核心判定可能误判 | 人物多、自动化优先的场景 |

**《妈，您人设崩了》案例选硬编码**：
- 47 个角色，主 Agent 已读完所有 11 part 摘要
- 描述包含"剧情关键节点"（如"夫前目犯成连城仕途助手"），自动统计无法做到
- 核心人物判定标准：影响 ≥ 2 个 part 主线 + 有完整人物弧光

**实测效果**：
- 23 个核心人物（加粗），24 个路人
- 按汉字 unicode 排序分组（拼音需要 pypinyin 库，不在 Python 标准库），分组效果接近拼音
- 读者读完全书摘要后可快速查阅"谁是谁"

## 主题 27：每本书一个独立项目目录（2026-06-10 沉淀）

**适用场景**：处理一本新小说时，所有产出文件（part 切分、part 摘要、进度笔记、脚本、汇总）的存放位置。

**经验沉淀**（2026-06-10，《妈，您人设崩了》用户反馈）：

**问题**：旧版 SKILL 的脚本示例（`split_novel2.py` / `merge_witch_summaries.py`）默认把所有文件散在源 txt 所在目录（如 `H:\搜书吧\`）。一本小说会产出约 25 个文件（11 个 part_NNN.txt + 11 个 part_NNN_summary.md + 进度笔记 + 汇总 + 脚本），全部堆在源目录会**严重污染**用户的小说库。

**用户原话**（2026-06-10）：
> "我发现你将产出内容放在了一个文件夹中，这个挺不错的，这样我的文件就不会乱了"

**解决方案**：处理新小说时，主 Agent **主动**在源 txt 同级创建独立项目目录：

```
H:\搜书吧\<小说名>.txt                       # 源文件（原位置）
H:\搜书吧\<小说名简化>_摘要\                  # 新建项目目录
  ├── split_<简化>.py                       # 切分脚本
  ├── merge_<简化>.py                       # 合并脚本
  ├── renshe_part_001.txt ~ part_NNN.txt    # 切分文件（中间产物，最后清理）
  ├── part_001_summary.md ~ part_NNN.md     # 子 Agent 输出的摘要
  ├── <书名>_进度笔记.md                     # 进度跟踪
  └── <书名>_全书摘要汇总.md                  # 最终合并产物
```

**目录命名规则**：
- 用书名简化形式（去标点、去副标题）+ `_摘要` 后缀
- 例：`妈，您人设崩了-全本 番外.txt` → `妈您人设崩了_摘要\`
- 例：`放开那个女巫 10.0改加料.txt` → `放开那个女巫_摘要\`
- 例：`烈火凤凰.txt` → `烈火凤凰_摘要\`

**实施方式**：

1. **派子 Agent 前**先建目录：

```bash
mkdir -p "H:/搜书吧/<书名简化>_摘要"
```

2. **所有脚本的 OUT_DIR 指向该目录**：

```python
WORK_DIR = r'H:\搜书吧'
OUT_DIR = os.path.join(WORK_DIR, '<书名简化>_摘要')

# 切分输出
for i, p in enumerate(parts):
    out_path = os.path.join(OUT_DIR, f'renshe_part_{i+1:03d}.txt')
    ...

# 子 Agent 提示词中的输出路径
prompt = f"""
**输出文件**：`{OUT_DIR}\\part_{n:03d}_summary.md`
"""

# 合并脚本扫描该目录
for f in sorted(os.listdir(OUT_DIR)):
    if re.match(r'part_\d+_summary\.md$', f):
        ...
```

3. **清理时只删 OUT_DIR 内的 renshe_part_*.txt**：

```bash
cd "H:/搜书吧/<书名简化>_摘要"
rm renshe_part_*.txt
```

**好处**：
- ✅ 用户的源小说目录不会被污染（一本书的产出全部隔离）
- ✅ 跨会话接续时，只需 `ls <书名简化>_摘要/` 就能看到所有产出
- ✅ 用户要归档/删除整本书的摘要时，直接删整个文件夹
- ✅ 多本书并行处理时彼此不冲突（每本一个目录）

**反例**（不要这样做）：
- ❌ 把所有产出散在源 txt 同级目录
- ❌ 用通用名（如 `summaries\`）做目录名——多本书会冲突
- ❌ 目录嵌套过深（如 `H:\搜书吧\摘要\妈您人设崩了\`）——增加路径复杂度

**与其他主题的关系**：
- 主题 5（清理 part 临时文件）：清理时只需 `rm <目录>/renshe_part_*.txt`，更安全
- 主题 7（进度笔记跨会话接续）：进度笔记放在项目目录内，新会话只需 `Read <目录>/<书名>_进度笔记.md`
- 主题 19（作者名文件名一致性）：目录名只用书名简化，不含作者名（避免重名作者带来的歧义）
